# 0004: Briefing caching with timestamp-based invalidation

- **Status:** Accepted
- **Date:** 2026-05-28
- **Deciders:** Tanmay Hatkar

## Context

Slice 4 introduced morning briefings: a natural-language summary of the user's day, generated by the LLM from current commitments. The first version was **stateless** — every `GET /briefings/today` re-ran the LLM, regenerating from scratch.

That had real downsides:

1. **Latency on every page load.** Each briefing request added ~500 ms of LLM wait. Opening the app in the morning, refreshing the tab, switching between tabs — all paid this cost.

2. **Wasteful LLM spend.** Even at gpt-4o-mini prices (~$0.0001 per briefing), generating identical content multiple times for the same day is just paying for the same answer over and over. At Groq's free tier the cost is zero, but the rate-limit budget still ticks.

3. **Inconsistent UX.** Because LLMs have temperature > 0 for prose, two identical requests produce slightly different briefings. A user refreshing the page would see the briefing subtly change for no apparent reason — felt flaky.

The fix: cache the briefing. But naive "cache for the whole day" would mean the briefing goes stale the moment any commitment changes (created, edited, completed). The user adds a new commitment via the form, the briefing should reflect it.

So we need cache + automatic invalidation.

## Decision

**Persist briefings keyed by date. Validate cache freshness on read by comparing the briefing's `generated_at` to the most recent `updated_at` across all commitments.**

### Schema

A new SQLite table:

```sql
CREATE TABLE briefings (
    id            TEXT PRIMARY KEY,
    date          TEXT NOT NULL UNIQUE,
    content       TEXT NOT NULL,
    today_count   INTEGER NOT NULL,
    overdue_count INTEGER NOT NULL,
    generated_at  TEXT NOT NULL
)
```

`UNIQUE(date)` enforces one briefing per day. Writes use `ON CONFLICT(date) DO UPDATE` (SQLite upsert) so regenerating overwrites instead of erroring.

### Read path

```python
def get_today(self, force_regenerate: bool = False) -> BriefingResponse:
    today = date.today()

    if not force_regenerate:
        cached = self._repo.get_for_date(today)
        if cached is not None and self._is_cache_fresh(cached):
            return cached  # cache hit, no LLM call

    return self._generate_and_save(today)

def _is_cache_fresh(self, cached) -> bool:
    latest = self._service.latest_commitment_update()
    if latest is None:
        return True  # no commitments at all = cache trivially fresh
    return cached.generated_at > latest  # STRICT comparison
```

### Why strict `>` not `>=`

If `cached.generated_at == latest` (same microsecond — possible on fast machines), we can't tell which event happened first. Safer to regenerate.

This caught a real test failure during implementation: the unit test created a commitment immediately after a briefing, the timestamps collided, the cache was considered fresh, the test failed. Switching to strict `>` made the test deterministic.

### Force regeneration

`GET /briefings/today?force_regenerate=true` bypasses the cache. The frontend's refresh button uses this so users have an explicit "give me a new one" option.

### Response shape

`BriefingResponse` gained a `cached: bool` field. The route returns this to the frontend so it could (in the future) display "cached" vs "fresh" indicators. Today the frontend doesn't surface this — but the data is there if we want.

## Alternatives considered

### No caching (the original slice 4 implementation)

Just regenerate every time. Simple, always fresh.

**Rejected because:** every page load = LLM call = noticeable latency. Especially annoying when refreshing the UI to test other features during development. The cost (engineering complexity of caching) was lower than the cost (paid by the user on every load).

### Time-based TTL (cache for N minutes)

"Cache valid for 30 minutes" or similar. Simpler than commitment-aware invalidation.

**Rejected because:** the cache could go stale immediately after a commitment change but stay valid for 30 minutes. User adds a commitment, briefing doesn't reflect it for half an hour. Bad UX.

### Cache invalidation via explicit hooks

Every commitment mutation (create/update/delete) explicitly invalidates the cache. CommitmentService gets a dependency on a "cache-busting" interface.

**Rejected because:**
- Couples CommitmentService to briefing concerns. Violates single responsibility.
- Easy to forget to invalidate from some new code path.
- Requires plumbing the briefing repo into the commitment service.

The timestamp-based approach keeps the dependency direction clean: BriefingService depends on CommitmentService (to read the latest update time). CommitmentService knows nothing about briefings.

### Event sourcing / mutation log

Maintain a log of all mutations. Cache invalidation reads from this log.

**Rejected because:** way too much machinery for slice 6. Useful at scale, not at our scale.

### Cache per-user instead of globally

For multi-user (which we don't have), each user would have their own cache key.

**Deferred:** when we add users, we add `user_id` to the briefings table and key the cache by `(user_id, date)`. Simple migration. Not relevant now.

## Consequences

### Positive

- **Fast page loads.** Cache hit = no LLM call = ~10ms response instead of ~500ms.
- **Lower LLM spend.** Per-day briefings now cost one LLM call per day per user, regardless of how many times they open the app.
- **Consistent UX.** The briefing doesn't subtly change every time you refresh.
- **Auto-invalidation.** Adding/editing/completing commitments naturally invalidates the cache — no manual cache-busting calls scattered through code.
- **Explicit override.** The refresh button gives users an escape hatch (force_regenerate=true).
- **Clean dependency direction.** BriefingService depends on CommitmentService; not the other way around.

### Negative

- **Doesn't catch deletions.** If a commitment is deleted, `updated_at` on the deleted row no longer exists. The remaining commitments' `updated_at` values are unchanged. The briefing might still mention the deleted commitment until something else triggers regeneration.

  **Workarounds the user has:** the refresh button (manual force), or doing any other commitment mutation (which bumps the latest timestamp).

  **Real fix (future):** soft deletes (mark `deleted_at` instead of removing rows) or a separate `commitments_changed_at` table that's bumped on any mutation including deletes.

- **Microsecond collisions.** Required strict `>` comparison instead of `>=`. Caught by a test; documented here.

- **Additional table to migrate when moving to Postgres.** Minor — same `CREATE TABLE` pattern, plus the `ON CONFLICT` upsert syntax differs slightly (Postgres uses `ON CONFLICT (date) DO UPDATE SET ... = EXCLUDED.*`, mostly compatible).

- **One more dependency in the BriefingService constructor.** It now takes `commitment_service` AND `briefing_repository`. Increases construction complexity at the route layer (`_build_briefing_service` chains four objects). Acceptable.

### Future considerations

- **Stale-cache fallback.** If a fresh LLM call fails (provider down), should we return the stale cache as a fallback instead of 503? Probably yes. Add later.
- **Cache history.** With briefings persisted, we could expose `GET /briefings?since=YYYY-MM-DD` to view past briefings. Could power a "what did I do last week" feature.
- **Per-user caching** when multi-user arrives (see above).
- **Compaction.** After months of use, the briefings table grows by one row per day. At 365 rows/year, this is nothing — but worth noting.
